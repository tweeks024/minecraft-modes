# Meteor Staff visual upgrade — design

**Goal:** Replace the placeholder 16×16 sprite + flat `item/handheld` model for `wildwest:meteor_staff` with a polished **3D voxel staff model** for the player's hand and a **dedicated 2D pixel-art icon** for the inventory/GUI. Ship a `.bbmodel` source so the 3D model is editable in Blockbench, and a Python generator so the assets can be regenerated deterministically.

**Non-goals:**
- No gameplay/behavior changes. [MeteorStaffItem.java](wildwest/src/main/java/com/tweeks/wildwest/item/MeteorStaffItem.java) is untouched.
- No animation rig, no glow shader, no custom particles on the held item. Magma "glow" is texture-only (bright orange UV regions that look emissive but use vanilla solid shading).
- No data-driven tinting or model overrides keyed off NBT/cooldown state. The model is static.

## Visual theme

**Charred-wood haft + magma core head** (locked during brainstorm).

- Diagonal scorched wooden shaft (dark brown grain with black streaks).
- Two iron bands wrap the shaft (low grip + mid).
- A glowing magma stone cluster sits at the staff head — central magma cube plus 4 short ember spikes.
- Two tiny ember "wisps" float just above the head for sparkle.

Palette:
- Charred wood: `#3a2412` base, `#1a0e08` grain streaks, `#5a3820` highlight.
- Iron bands: `#4a4540` base, `#707070` rivet dots, `#2a2520` shadow.
- Magma core: `#ff5a1f` outer, `#ffb02e` hot center, black crack lines.
- Ember spikes / wisps: `#ff8533`, `#ffd066` hot tip.

## Architecture

Minecraft 1.21.4+ (NeoForge 26.x) ships the **ItemModelDefinition** system. The file at `assets/wildwest/items/meteor_staff.json` can branch on `display_context` to route the GUI display context to a different model than the in-hand contexts.

```
items/meteor_staff.json            ← ItemModelDefinition (selector)
  ├─ display_context=gui     → models/item/meteor_staff_gui.json (flat 2D)
  └─ default                 → models/item/meteor_staff.json     (3D voxel)
```

This means the 3D model's `display.gui` transform is unused (the selector intercepts before display transforms apply), so the 3D model can be tuned purely for hand feel without GUI-readability compromises.

### Files

| Path | Purpose | Authoring |
|------|---------|-----------|
| `wildwest/src/main/resources/assets/wildwest/items/meteor_staff.json` | `display_context` selector | Hand-written, small JSON |
| `wildwest/src/main/resources/assets/wildwest/models/item/meteor_staff.json` | 3D voxel model (elements + display transforms) | Generated by `gen_meteor_staff.py`, can be overwritten by Blockbench export from `.bbmodel` |
| `wildwest/src/main/resources/assets/wildwest/models/item/meteor_staff_gui.json` | Flat 2D inventory icon, parent `item/generated` | Generated by `gen_meteor_staff.py` |

## 3D voxel model

**Coord space:** Minecraft item models use a 16×16×16 voxel space, origin at (0,0,0), opposite corner (16,16,16). All cube `from`/`to` are in this space.

**Shaft (diagonal):** 7 stacked cubes climbing from lower-front to upper-back. Each cube is 2×2×2 voxels:

| Cube | from | to |
|------|------|-----|
| shaft_0 | (4.0, 1.0, 8.0) | (6.0, 3.0, 10.0) |
| shaft_1 | (4.5, 3.0, 8.0) | (6.5, 5.0, 10.0) |
| shaft_2 | (5.0, 5.0, 8.0) | (7.0, 7.0, 10.0) |
| shaft_3 | (5.5, 7.0, 8.0) | (7.5, 9.0, 10.0) |
| shaft_4 | (6.0, 9.0, 8.0) | (8.0, 11.0, 10.0) |
| shaft_5 | (6.5, 11.0, 8.0) | (8.5, 13.0, 10.0) |
| shaft_6 | (7.0, 13.0, 8.0) | (9.0, 15.0, 10.0) |

This produces a visible diagonal rise without true rotation (model JSON only allows axis-aligned rotation per element, and we want to keep the bbmodel-editable as plain cubes).

**Iron bands:** 2 cubes, 1-voxel thick wraps around the shaft, slightly oversized:

| Cube | from | to |
|------|------|-----|
| band_low | (3.5, 2.5, 7.5) | (6.5, 3.5, 10.5) |
| band_mid | (5.5, 8.5, 7.5) | (8.5, 9.5, 10.5) |

**Magma core (head):** central 3×3×3 magma cube + 4 small 1-voxel spike cubes radiating from it:

| Cube | from | to |
|------|------|-----|
| core_main | (6.5, 14.5, 7.5) | (9.5, 17.5, 10.5) |
| spike_top | (7.5, 17.5, 8.5) | (8.5, 18.5, 9.5) |
| spike_front | (7.5, 15.5, 6.5) | (8.5, 16.5, 7.5) |
| spike_back | (7.5, 15.5, 10.5) | (8.5, 16.5, 11.5) |
| spike_right | (5.5, 15.5, 8.5) | (6.5, 16.5, 9.5) |

Some `to` coordinates exceed 16 (`to.y` up to 20, `to.x` up to 11.5). Minecraft permits element coordinates in the range `-16` to `32` for items without renderer warnings or aggressive culling, so we are well within safe margins.

**Ember wisps:** 2 tiny 1×1×1 floaters above the head:

| Cube | from | to |
|------|------|-----|
| wisp_a | (6.0, 19.0, 9.0) | (7.0, 20.0, 10.0) |
| wisp_b | (8.5, 18.5, 8.0) | (9.5, 19.5, 9.0) |

**Total: 16 cubes.**

## Display transforms

Tuned so the staff reads as a real diagonal weapon in hand. Values follow the same conventions as vanilla `item/handheld` but customized:

```json
"display": {
  "thirdperson_righthand": { "rotation": [0, 90, -35], "translation": [0, 1.25, 1.5], "scale": [0.85, 0.85, 0.85] },
  "thirdperson_lefthand":  { "rotation": [0, 90, -35], "translation": [0, 1.25, 1.5], "scale": [0.85, 0.85, 0.85] },
  "firstperson_righthand": { "rotation": [0, -90, 25], "translation": [1.13, 3.2, 1.13], "scale": [0.68, 0.68, 0.68] },
  "firstperson_lefthand":  { "rotation": [0, -90, 25], "translation": [1.13, 3.2, 1.13], "scale": [0.68, 0.68, 0.68] },
  "ground":                { "rotation": [0, 0, 0], "translation": [0, 2, 0], "scale": [0.5, 0.5, 0.5] },
  "fixed":                 { "rotation": [0, 0, 0], "translation": [0, 0, 0], "scale": [1.0, 1.0, 1.0] },
  "gui":                   { "rotation": [0, 0, 0], "translation": [0, 0, 0], "scale": [1.0, 1.0, 1.0] },
  "head":                  { "rotation": [0, 180, 0], "translation": [0, 13, 7], "scale": [1.0, 1.0, 1.0] }
}
```

`gui` is unused in practice because the selector routes GUI to the 2D model, but we ship a default so the model is still well-formed if it's ever rendered standalone in dev.

## Model JSON texture binding

Item element models reference textures via *variable* names defined in a `textures` block, not direct namespaced paths. The 3D model JSON must include:

```json
{
  "textures": {
    "0": "wildwest:item/meteor_staff",
    "particle": "wildwest:item/meteor_staff"
  },
  "elements": [
    {
      "from": [4.0, 1.0, 8.0], "to": [6.0, 3.0, 10.0],
      "faces": {
        "north": { "uv": [0, 0, 2, 2], "texture": "#0" },
        ...
      }
    }
  ]
}
```

Every face must reference `"#0"`. The `particle` entry is what Minecraft uses when the item is dropped and shatters into break particles — without it you get the missing-texture pink/black square. The 2D GUI model uses the standard `layer0` slot inherited from `item/generated`.

## Texture layout

### 3D texture (`meteor_staff.png`, 32×32)

UV regions packed by hand with a **1-pixel margin between groups** to prevent mipmap bleeding. At low MIP levels the GPU samples averaged texels across UV boundaries; without a margin the cool-gray iron band pixels can bleed into adjacent warm magma pixels (and vice versa), producing dirty gray-orange seams at distance. Vanilla Minecraft textures use the same 1px-margin convention for the same reason.

The generator places per-cube faces into the atlas:

- **(0, 0) – (7, 23)** — shaft wood faces, 7×24 region (each shaft cube is geometrically identical; we use one set of side faces tiled).
- **(9, 0) – (15, 7)** — iron band faces (shared by both bands). 1-pixel gap at column 8.
- **(0, 25) – (31, 31)** — magma core 6-face wrap, full atlas-width strip. 1-pixel gap at row 24.
- **(17, 0) – (23, 7)** — ember spike + wisp faces (4 spikes + 2 wisps share one tiny atlas square). 1-pixel gap at column 16.

Margin pixels are filled with the dominant adjacent palette colour so that any bleed at distance is invisible (e.g. the column between shaft-wood and iron-band is filled with `#1a0e08`, the shared dark wood streak colour).

Each face is referenced by `uv: [u1, v1, u2, v2]` in the cube definition. Generator writes coordinates atomically with the cubes so they cannot drift out of sync.

### GUI icon (`meteor_staff_gui.png`, 16×16)

Hand-composed pixel art (drawn by the script, not a render):

- Diagonal shaft: 1-pixel-wide Bresenham line from `(3,14)` to `(11,5)` in `#3a2412`, plus a parallel `#1a0e08` shadow line offset 1 pixel down-left.
- Iron band 1 at `(4,12)–(6,13)` (3×2 px block in `#4a4540`, 1 rivet pixel in `#707070`).
- Iron band 2 at `(7,9)–(9,10)` (same construction).
- Magma blob: filled 3×3 square at `(11,3)–(13,5)` in `#ff5a1f`, with one `#ffb02e` highlight pixel at `(12,4)` and one black crack pixel at `(13,3)`.
- Glow halo: 4 dim `#ff8533` pixels at the diagonal corners `(10,2)`, `(14,2)`, `(10,6)`, `(14,6)`.
- 1 wisp pixel at `(13,1)` in `#ffd066`.

Result: a clear diagonal staff icon legible at 16×16 in a hotbar.

## .bbmodel format

Mirror the structure of [herobrine.bbmodel](wildwest/tools/herobrine.bbmodel) — JSON with:

```jsonc
{
  "meta": { "format_version": "4.5", "model_format": "java_block" /* item models use this */ },
  "name": "meteor_staff",
  "model_identifier": "wildwest:meteor_staff",
  "visible_box": [1, 1, 0],
  "resolution": { "width": 32, "height": 32 },
  "elements": [ /* 16 cube objects, each with from/to/uv per face/origin/uuid */ ],
  "outliner": [ "<uuid>", "<uuid>", ... ],   /* flat element list, no bones */
  "textures": [ { "name": "meteor_staff", "source": "data:image/png;base64,...", "uuid": "<uuid>", ... } ],
  "display": { /* mirrors the model JSON display block */ }
}
```

Use `java_block` `model_format` — Blockbench uses this value for the "Java Block/Item" project type that exports to `models/item/*.json`. (The mob bbmodels in [tools/](wildwest/tools/) use `modded_entity` instead because they export to entity-model Java classes, not item models — different pipeline.) Embed the 32×32 PNG as base64 so the file is self-contained. Use deterministic UUIDs (`uuid.uuid5` with a fixed namespace) so re-running the generator produces byte-identical output.

When the user opens the bbmodel, edits cubes/UV/textures, and exports as "Java Item Model" → that overwrites `models/item/meteor_staff.json`. The bbmodel itself stays the source of truth for further edits.

## Generation script

`wildwest/tools/gen_meteor_staff.py`:

```python
def main():
    out_dir = sys.argv[1] if len(sys.argv) > 1 else "wildwest/tools/"
    cubes = build_cubes()          # returns list of (name, from, to, uv_map, color_group)
    texture_3d = paint_3d_texture(cubes)   # 32x32 PIL image
    texture_gui = paint_gui_icon()         # 16x16 PIL image
    bbmodel = build_bbmodel(cubes, texture_3d)
    model_3d = build_model_json(cubes)
    model_gui = build_gui_model_json()

    write(out_dir + "meteor_staff.bbmodel", json.dumps(bbmodel, indent=2))
    write(ASSETS + "models/item/meteor_staff.json", json.dumps(model_3d, indent=2))
    write(ASSETS + "models/item/meteor_staff_gui.json", json.dumps(model_gui, indent=2))
    texture_3d.save(ASSETS + "textures/item/meteor_staff.png")
    texture_gui.save(ASSETS + "textures/item/meteor_staff_gui.png")
```

Dependencies: stdlib + Pillow (already used by [gen_textures.py](wildwest/tools/gen_textures.py)).

## ItemModelDefinition selector

Rewrite `assets/wildwest/items/meteor_staff.json` to use a `select` on `display_context`:

```json
{
  "model": {
    "type": "minecraft:select",
    "property": "minecraft:display_context",
    "cases": [
      { "when": "gui", "model": { "type": "minecraft:model", "model": "wildwest:item/meteor_staff_gui" } }
    ],
    "fallback": { "type": "minecraft:model", "model": "wildwest:item/meteor_staff" }
  }
}
```

The selector value `gui` matches the same context Minecraft uses when rendering inventory slots, item frame icons in maps, recipe books, etc. All other contexts (held, ground, head) fall through to the 3D model.

## Testing

Visual-only feature — no unit tests. Verification:

1. `./gradlew :wildwest:runData` (datagen runs cleanly, no model validation errors).
2. `./gradlew :wildwest:runClient` — spawn a Meteor Staff via `/give @s wildwest:meteor_staff`.
   - Inventory hotbar slot shows the 2D pixel-art icon.
   - First-person view shows the 3D voxel staff at the documented diagonal.
   - Third-person view shows the 3D staff in the player's hand.
   - Drop on ground → 3D model lying flat, scale 0.5.
   - Place in item frame → 3D model fills the frame.
3. Open `wildwest/tools/meteor_staff.bbmodel` in Blockbench → all cubes visible, texture maps correctly, display transforms previewable.
4. PNG byte sanity: `file textures/item/meteor_staff.png` reports `32 x 32`, `meteor_staff_gui.png` reports `16 x 16`.

## Risks / open questions

- **Cube count vs perf:** 16 cubes per item is well under any practical limit (vanilla shield has more). Negligible.
- **Hand z-fighting:** Magma spike cubes that extend past y=16 may clip into the player's view slightly in first-person at high FOVs. Mitigation: the `firstperson_righthand` translation `[1.13, 3.2, 1.13]` pulls the model forward and right, putting the magma well clear of the view frustum. If the user reports clipping during smoke test, adjust `translation.y` upward.
- **Ground-context float:** Because the `display_context=gui` selector intercepts only the GUI render path, the `ground` context still uses the 3D model. The model's coordinate origin is at the base of the staff (y=1), so `ground.translation = [0, 2, 0]` may leave the dropped item floating a couple of voxels above the dirt at scale 0.5. Smoke-test confirms whether to lower to `[0, 0, 0]` or `[0, -1, 0]`.
- **`java_block` vs `java_item` model_format:** Blockbench historically called both project types `java_block` internally; if a newer Blockbench version splits them, we may need `model_format: "java_item"`. Validate by opening the generated bbmodel in Blockbench during the manual smoke test. If it loads as a block project, swap the value and regenerate.
- **GUI icon at small scales:** The 16×16 icon must remain legible. The design pushes the magma to the upper-right and shaft along the diagonal, mimicking vanilla sword/trident icon conventions. If it reads as a blob at a glance, the script's paint function is the single place to iterate.
